Exploring Layers and Ripples: Studying English with Computer Parsing

Alastair Butler

Hirosaki University



This book introduces computer parsing of English, with an emphasis on how this can help with the goal of studying English.

    Once a word occurs, ripple effects start that connect following words to form an utterance. Ripple effects place constraints on what following words might be and how they can be used. We can model ripple effects by supposing that words appear under layers that build up a connected tree structure for all the content of a single sentence. Parsing language is the activity of identifying layers of structure while obeying the consequences of ripple effects.

    The book provides Definite Clause Grammar (DCG) definitions that are executable code for the XSB Tabling Prolog system. These definitions create an automatic parsing system with a wide coverage of English. The system gives feedback about the acceptability and also use of the English that we try to parse.

Chapter 1
Introduction


1.1   Encircling layers

An utterance of English is like an onion with encircling layers around internal elements, as pictured in Figure 1.1.

Figure 1.1: Layers around an innermost element

   

The action of PARSING amounts to:

(i)
pealing back layers to reach internal elements, and then
(ii)
placing everything back together again.

An innermost element is either the head word of a phrase layer or the verb of a clause layer. While placing things back together again, we can add labels called annotations to keep track of what was uncovered when we pealed back the layers. Subsequently, these labels act as handles for accessing the structural analysis revealed by parsing. For the purpose of recording parse analysis, this book adopts the annotation scheme of the Treebank Semantics Parsed Corpus (TSPC; Butler 2022). The TSPC is a corpus of English for general use with hand worked analysis for approaching half-a-million words.

    Furthering the onion analogy, the kind of layering seen in Figure 1.2 occurs when there is coordination, with each conjunct having to conform to whatever is possible at the same particular layer.

Figure 1.2: Parallel layers

   
   

    In addition to structures that arise from multiple layers, we will observe ripple effects, where what happens at one layer goes on to have consequences for what is possible in other layers.


1.2   Phrase structure rules

Phrase structure rules are a way to build labelled layers around embedded elements. A phrase structure rule has:

(i)
a left-hand side that is the label for the layer
(ii)
a right-hand side made of one, two, or more embedded elements, where an embedded element can be either the label of a layer or a terminal element

    For example, the five rules of (1.1) involve labelled layers: vp, np_sbj, and ip; and terminal elements: [smiles], [and], and [he].

(1.1)
1.  vp --> [smiles].
2.  vp --> vp, [and], vp.
3.  np_sbj --> [he].
4.  ip --> np_sbj, vp.
5.  ip --> ip, [and], ip.

By following the rules of (1.1), we can assemble the structure of (1.2): We can start with rule 4 where an ip layer can consist of an np_sbj layer followed by a vp layer. With rule 3 we get to a complete np_sbj with the terminal word [he]. With rule 2 we get to a vp, which is itself made up of a vp, followed by an [and] word, followed by a vp. Finally, with rule 1 we can get to complete vp layers with the terminal word [smiles].

1.3   Orientation

Computer parsing in this book will use Definite Clause Grammar notation (DCG; Pereira and Warren 1980). This is a notation for writing phrase structure grammar rules in which labels for layers take attributes with values unified by Prolog-style term unification. DCG notation is a feature of virtually all Prolog systems. With DCG notation, we can write a set of phrase structure rules and get an executable Prolog program.

    We will want to write phrase structure grammar rules with left recursion, like rules 2 and 5 of (1.1). When there is left recursion, avoiding infinite loops from rule evaluation requires remembering what has already been evaluated. This is possible with tabling, a feature of some Prolog systems, notably, the XSB Tabling Prolog system (Swift and Warren 2022).

    We will proceed as follows. The remainder of this chapter, section 1.4, introduces rules for utterance layers. This will introduce both writing DCG rules and using the Prolog system to query English word input. Subsequent chapters introduce rules to build up a wide coverage of consequences from word input. This starts in chapter 2 with an inventory of all the kinds of words there are in English. Once we have a handle on the range of words, we will want to start putting words together to create phrases, and this is the goal of chapter 3. With access to phrases and words, we will have content for clause layers. Establishing what is possible at the clause layer is the goal of chapter 4. Chapter 5 builds on all the preceding chapters by integrating clause subordination.


1.4   Utterance layers

The utterance layer is the topmost structural layer of an utterance. The Prolog code of (1.4) defines this layer with a series of phrase structure rules, each of which imposes its own conditions on what needs to be present to say there is content for an utterance layer.

(1.4)
utterance(Id,n(Id,IP,none)) -->
  clause_top_layer(statement,[],IPl-L1),
  punc(final,L1-[]),
  {
    IP =.. ['IP-MAT'|IPl]
  }.
utterance(Id,n(Id,IP,none)) -->
  verb_phrase_layer([],filled_subject,imperative,active,IPl-L1),
  punc(final,L1-[]),
  {
    IP =.. ['IP-IMP'|IPl]
  }.
utterance(Id,n(Id,'CP-QUE'(IP,PU),none)) -->
  clause_top_layer(matrix_interrogative,[],IPl-[]),
  punc(final_question,[PU]-[]),
  {
    IP =.. ['IP-SUB'|IPl]
  }.
utterance(Id,n(Id,IP,none)) -->
  clause_top_layer(_Type,[],IPl-[]),
  {
    IP =.. ['IML'|IPl]
  }.
utterance(Id,n(Id,'FRAG'(NP),none)) -->
  noun_phrase('',[NP]-[]).
utterance(Id,n(Id,'FRAG'(ADJP),none)) -->
  adjective_phrase('',[ADJP]-[]).
utterance(Id,n(Id,'FRAG'(Phrase),none)) -->
  adverbial([Phrase]-[]).

    The first rule of (1.4) will create a structure with IP-MAT as the topmost node, given that the content to be parsed involves a list of items where all but the last item satisfies one of the clause_top_layer phrase structure rules (see section 4.2 below), and where the last list item is an instance of final punctuation (see section 2.12 below).

    In Prolog, lists are written with square bracket notation: [a1, ..., aN], where a1 through to aN are the list items.

    In (1.5), a series of Prolog calls are made. The first call has tphrase_set_string to establish a list of items to parse. Let's call this the parse list. The second call has parse to question whether the established parse list has content to satisfy the utterance rules of (1.4). If parse succeeds then parse results are pretty printed as bracketed tree output.

(1.5)
| ?- tphrase_set_string(['PRO'('He'),'VBP;~I'(smiles),'PU'('.')]), parse(utterance('ex1.5')).

( (IP-MAT (NP-SBJ (PRO He))
          (VBP;~I smiles)
          (PU .))
  (ID ex1.5))

yes

The pretty print from (1.5) shows the return of a structure with IP-MAT (declarative matrix clause) as the topmost node, from which it follows that the first rule of (1.4) completed successfully.

    Now consider (1.6). This involves almost the same parse list as (1.5), only with the last list item (the punctuation: 'PU'('.')) absent.

(1.6)
| ?- tphrase_set_string(['PRO'('he'),'VBP;~I'(smiles)]), parse(utterance('ex1.6')).

( (IML (NP-SBJ (PRO he))
       (VBP;~I smiles))
  (ID ex1.6))

yes

The pretty print from (1.6) shows the return of a structure with IML (intermediate clause layer) as the topmost node: The first three rules of (1.4) will fail, but the fourth rule succeeds. The fourth rule is very similar to the first rule: Both call the clause_top_layer phrase structure rules, but the fourth rule sends all of the parse list content to its clause_top_layer call, and so lacks the requirement for the last item of the parse list to be sentence final punctuation.

    As a further example of checking if there is parse list content for an utterance layer, consider (1.7).

(1.7)
| ?- tphrase_set_string(['NS'('boys')]), parse(utterance('ex1.7')).

( (FRAG (NP (N boys)))
  (ID ex1.7))

yes

The pretty print from (1.7) shows the return of a structure with FRAG (sentence fragment) as the topmost node. This will have been achieved with the fifth rule of (1.4), which expects the parse list to provide content for the success of one of the noun_phrase rules of section 3.2 below.

    To sum up, this section has shown how to query whether a parse list contains content for an English utterance. With success, returned structure tells us about the kind of utterance: declarative matrix sentence (IP-MAT), imperative sentence (IP-IMP), interrogative sentence (CP-QUE), intermediate clause utterance (IML), or sentence fragment (FRAG) comprised of a single noun phrase, a single adjective phrase, or a single adverbial phrase.





Chapter 2
Words


2.1   Introduction

This chapter catalogs all of the types of words we will aim to distinguish. This is achieved by saying that a word belongs to a particular WORD CLASS. Also some words, notably verbs, take a grammatical code that is additional to their word class information. As seen in the queries of section 1.4, we capture the parse consequences for a word by having it as an item in a parse list together with its word class and any grammatical code. The work of this chapter is to establish the initial parse outcome of encountering a class(code)/word pairing in a parse list.


2.2   Noun words

We start off with the noun rules of (2.1) for establishing the contribution of the general noun words of Table 2.1.

Table 2.1: Tags for general noun words

2.8   Verb words

This section covers verb words. The verb rule of (2.13) constructs verb words with labels created from two parameters: Tag and Code.

(2.13)
verb(Tag,Code,[TagCodeWord|L]-L) -->
  [TagCodeWord],
  {
    atom_concat(Tag,Code,TagCode),
    TagCodeWord =.. [TagCode,Word]
  }.

Full integration of a verb is realised by verb_with_complement of (2.14). This combines a tag from verb_tag of (2.15) below consistent with the inflection information of the Infl parameter together with a compatible code from verb_code of (2.16), to then:

(i)
construct a verb by calling the verb construction rule of (2.13)
(ii)
establish the verb's complement selection requirements through a call of verb_complement (from section 2.8.2 below)
(2.14)
verb_with_complement(Displaced,Sbj_type,Infl,Voice,L-L0) -->
  {
    verb_tag(Infl,Tag_list),
    member(Tag,Tag_list),
    sub_atom(Tag,0,1,_,C),
    verb_code(C,Code)
  },
  verb(Tag,Code,L-L1),
  verb_complement(Tag,Code,Displaced,Sbj_type,Voice,L1-L0).

In the remainder of this section: Section 2.8.1 distinguishes different verb classes with tags that vary to mark form and inflection information. Section 2.8.2 distinguishes different grammatical codes for verbs based on consequences for complement selection.

2.8.2   Verb codes

We also provide verb codes as tag label extensions in order to further distinguish verbs based on the selection criteria each verb has for its complements. The verb code system for main verbs that we will adopt was developed for Cowie (1989), where the codes are associated with word sense definitions. The Cowie (1989) system is a mnemonic based reworking of the earlier system of Hornby (1975).

    The different classes of verb allow for different verb codes. We obtain compatible verb codes from calls to verb_code of (2.16), which takes a capital letter as the value for its first parameter to identify the verb class: V for lexical verbs, D for DO verbs, H for HAVE verbs, or B for BE verbs. A verb_code call will then return through the Code parameter a code picked from the corresponding list for compatible codes.

(2.16)
verb_code('H',Code) :-
  member(Code,[';~Tn',';~cat_Vt',';~cat_Ve']).
verb_code('B',Code) :-
  member(Code,[
               ';~La',';~Ln',
               ';~I',';~Ip',';~Ipr',
               ';~cat_Vt',';~cat_Vg',';~cat_Ve',
               ';~ex_V',';~ex_Vp',';~ex_Vpr',';~ex_cat_Vt',';~ex_cat_Vg',';~ex_cat_Ve',
               ';~equ_Vf',';~equ_Vw',';~equ_Vt',';~equ_Vg',
               ';~cleft_Vn'
              ]).
verb_code('D',Code) :-
  member(Code,[';~I',';~Tn']).
verb_code('V',Code) :-
  member(Code,[
               ';~La',';~Ln',
               ';~I',';~Ip',';~Ipr',';~It',
               ';~Tn',';~Tn.p',';~Tn.pr',';~Tf',';~Tw',';~Tt',';~Tnt',';~Tni',';~Tg',';~Tng',
               ';~Dn.n',';~Dn.pr',';~Dn.f',';~Dn.w',';~Dn.t',
               ';~Cn.a',';~Cn.n',';~Cn.t',';~Cn.i',';~Cn.g',
               ';~cat_Vt',';~cat_Vg',';~cat_Ve'
              ]).

    Having determined the verb class and the relevant verb code we are in a position to state the complement consequences that result. This is accomplished with the verb_complement rules of sections 2.8.2.1–2.8.2.8.

2.8.2.1   Linking verbs

In addition to taking a filled subject, a linking verb requires a phrase with subject predicative function as its complement. This subject predicative can be either an adjective phrase or a noun phrase. This leads to the verb_complement rules of:

  • (2.17) when the verb code is ;~La and the subject predicative is an adjective phrase
(2.17)
verb_complement(_Tag,';~La',[],filled_subject,active,L) -->
  adjective_phrase('-PRD2',L).
  • (2.18) when the verb code is ;~Ln and the subject predicative is a noun phrase
(2.18)
verb_complement(_Tag,';~Ln',[],filled_subject,active,L) -->
  noun_phrase('-PRD2',L).
verb_complement(_Tag,';~Ln',[np(ICH)],filled_subject,active,['NP-PRD2'(ICH)|L]-L) -->
  [].

Chapter 3
Phrase layers


3.1   Introduction

This chapter considers phrase layers. These are intermediate levels of organisation between a word and the clause. Each phrase is made up of a HEAD word and any DEPENDENTs. The head word is the word that the phrase is required to have in order to exist, and may be the only element of the phrase. A phrase is named after the word class of the head word. There are four main phrase types:

  • noun phrases — with a noun head word (section 3.2)
  • adverb phrases — with an adverb head word (section 3.3)
  • adjective phrases — with an adjective head word (section 3.4)
  • preposition phrases — with a preposition head word (section 3.5)

3.2   Noun phrases

This section considers noun phrases. A noun phrase is created with (3.1) which calls noun_phrase_top to create list content (NPl). Operation NP =.. [Label|NPl] turns the collected elements of list NPl into daughter elements for a tree structure represented with a Prolog structure that has principal functor Label, consisting of 'NP' with an extension inherited from Ext. This assembled structure is added to a difference list that is a collection point for content outside the noun phrase.

(3.1)
noun_phrase(Ext,[NP|L]-L) -->
  noun_phrase_top(NPl-[]),
  {
    atom_concat('NP',Ext,Label),
    NP =.. [Label|NPl]
  }.

    A call of noun_phrase_top, (3.2), concerns a difference list (captured with L) that gathers content from either a call of noun_phrase_initial_layer (from section 3.2.1 below), or a call of noun_head_full (from section 2.2) which identifies words that essentially contribute a full noun phrase by themselves.

(3.2)
noun_phrase_top(L) -->
  noun_phrase_initial_layer(L).
noun_phrase_top(L) -->
  noun_head_full(L).

4.2   Basic clause structure

The top of a clause is established with the clause_top_layer rules of (4.1). Non-recursive rules of (4.1) all involve the introduction of the clause subject, each by a call to subject (from section 2.5). These non-recursive rules vary on where the subject occurs in the parse list, that is, either:

(i)
as an initial portion of the parse list, which leaves the clause Type parameter unspecified
(ii)
as a portion of the parse list that occurs immediately after the components discussed in section 4.3 below of an operator_layer or finite_have_be_or_mod_cat_layer, which requires the clause Type parameter to be set as matrix_interrogative

There are also recursive rules to enable:

(iii)
clause initial or clause final adverbials
(iv)
clause initial noun phrases or preposition phrases that are displaced items
(v)
conjuncts of coordinate structure, with conjuncts subsequent to the first gathered by calls to clause_top_tail of (4.2)

Adverbials for (iii) are picked up with adverbial calls (from section 5.5) that find either adverb phrases or the subordinate component of subordinate conjunctions. Displaced noun phrases and preposition phrases picked up with (iv) receive an index coindexed with an *ICH* (interpret constituent here marker) that is added to the Displaced parameter to ultimately be inherited as a selected item for a verb_complement call from section 2.8.2.

4.3   Matrix interrogatives

A matrix interrogative is formed by having an initial clause component that is either:

(i)
a modal, or a finite DO
(ii)
a finite HAVE, a finite BE, or a modal catenative
(iii)
a noun phrase that includes an initial word tagged WPRO, WPRO;_genm_, or WD
(iv)
an adverb phrase with an initial WADV word

A clause with (i) or (ii) is called a yes/no question. A clause with (iii) or (iv) is called a constituent question. For matrix constituent questions where the initial clause component under question is a non-subject, before the clause subject can occur, there needs to be either a modal, a finite DO, a finite HAVE, a finite BE. or a modal catenative.

    When either a modal, a finite DO, a finite HAVE, a finite BE, or a modal catenative occurs, it may be immediately followed by the contracted form of negation n't. The remainder of this section will add rules for matching these components, including the option for negation, so that they can be integrated into a parse with calls of clause_top_layer from section 4.2 above.

4.4   Clause structure with a main verb and prior catenative verbs

So far we have seen clause structure with only a single verb, but a clause might include a sequence of verbs. The MAIN VERB is either the only verb of a clause or the last verb to occur in a sequence of verbs of the same clause. Verbs of a verb sequence that are prior to the main verb are called CATENATIVE VERBS.

    Section 2.8.2.6 provides verb_complement rules for catenative verbs and now we focus on the clause structure needed to support those rules, with:

  • section 4.4.1 for code ;~cat_Vt with its selection of a to-infinitive clause as its complement
  • section 4.4.2 for code ;~cat_Ve with its selection of a past participle (-en) clause as its complement
  • section 4.4.3 for code ;~cat_Vg with its selection of a present participle (-ing) clause as its complement
  • section 4.4.4 for code ;~cat_Ve with its selection of a passive participle (-en) clause as its complement

4.4.1   Catenative verbs with to-infinitive complements

The verb_complement rule of section 2.8.2.6 for code ;~cat_Vt involves selection of a to-infinitive clause as its complement.

    The ip_to_inf rule of (4.21) will find content for a to-infinitive clause with a call to to_inf_layer of (4.22) for overall structure labelled IP-INF, possibly with an extension from the EXT parameter.

(4.21)
ip_to_inf(Ext,Displaced,Sbj_type,[IP|L]-L) -->
  to_inf_layer(Displaced,Sbj_type,VPl-[]),
  {
    atom_concat('IP-INF',Ext,Label),
    IP =.. [Label|VPl]
  }.

The first rule for to_inf_layer of (4.22) is non-recursive and picks up from the parse list an initial to word (see section 2.10) to then call verb_phrase_layer with the clause in a state of being infinitive and active. The other rules of to_inf_layer are recursive, so the parse will:

  • (the second rule) pick up an initial instance of a connective particle (which is always for) followed by content for a noun phrase that is given subject function; note that this rule can only occur when the Sbj_type parameter is unifiable to the unfilled_subject value, and that the value passed on for the Sbj_type parameter is filled_subject
  • (the third rule) pick up initial adverb phrases with a call to adverb_phrase (from section 3.3),
  • (the fourth rule) start a conjunction structure where all conjuncts are added with to_inf_layer with conjuncts subsequent to the first gathered by calls to to_inf_tail of (4.23).

Chapter 5
Clause subordination


5.1   Introduction

This chapter is concerned with integrating clause subordination into a parse. Clauses can be subordinate in virtue of having an argument role within a containing clause, such as subject (-SBJ) or object (-OB1). Section 5.2 considers that-clauses, which are finite clauses that typically function with an argument role in a containing clause. Section 5.3 provides coverage of embedded questions, which also typically function with an argument role in a containing clause.

    When a subordinate clause has a subject role within the containing clause, it will often occur as a notional subject extraposed to the right edge of the containing clause and in co-occurrence with a provisional subject it of the containing clause. Section 5.4 notes how parsing can capture the ripple effect connecting the provisional subject it and the extraposed notional subject.

    A clause can also be the subordinate component of a subordinate conjunction, which is covered in section 5.5. In section 5.6, subordination is also seen to occur with relative clauses, where a clause possibly with an initial relative pronoun and with an absence elsewhere of a selected clause item acts as a modifier. Section 5.7 adds support for cleft constructions.


5.2   CP-THT (that clause)

A that-clause is an embedded clause with a finite verb. This may have the complementizer that (tagged C) as the clause initial word. A that clause constructed with the first cp_that rule of (5.1) has a maximal constituent with the CP-THT tag that contains as the only element a finite clause layer (IP-SUB), which in turn contains all the clause content gathered with a clause_that_layer rule of (5.3), including any clause initial complementizer. The CP-THT tag can take function information as a tag extension provided with the Ext parameter of cp_that. If there is any conjunction at the highest level of a that clause, then this happens inside the IP-SUB layer within an IML (clause intermediate) layer constructed with the second cp_that rule of (5.1), which in turn calls recursive cp_that_tail of (5.2), which through additional calls of clause_that_layer picks up from the parse list subsequent content for conjuncts.

5.3   CP-QUE projections that are embedded question clauses

An embedded question clause is formed by having an initial clause component that is either:

(i)
an embedded question clause marker, that is, an instance of whether or if tagged WQ
(ii)
a noun phrase that includes an initial word tagged WPRO, WPRO;_genm_, or WD
(iii)
an adverb phrase with an initial WADV word

A clause with (i) is a yes/no question. A clause with (ii) or (iii) is a constituent question. In addition to the possibilities of (i)–(iii), internally an embedded question clause can be either:

(a)
a finite clause with statement word order (IP-SUB)
(b)
a to-infinitive clause (IP-INF)

The internal clause will be missing a selected item when a constituent question of case (ii).

    The maximal constituent of an embedded question clause has the CP-QUE tag, possibly with an extension from the EXT parameter taken by cp_que of (5.6). Calls of cp_que can involve:

(a)
a call of clause_que_finite_top_layer to gather under an immediately embedded IP-SUB projection the content of either (i) a finite clause with statement word order (the first rule) or (ii) the first conjunct of a conjunction of statement-word-order finite clauses with a recursive call to cp_que_finite_tail to pick up subsequent conjuncts (the second rule)
(b)
a call of clause_que_to_infinitive_top_layer to gather under an immediately embedded IP-INF projection the content of either (i) a to-infinitive clause (the third rule) or (ii) the first conjunct of a conjunction of to-infinitive clauses with a recursive call to cp_que_to_infinitive_tail to pick up subsequent conjuncts (the fourth rule)

5.5   Subordinate conjunction

This section focuses on rules able to integrate the subordinate clause component of subordinate conjunctions. These subordinate clauses can reside in containing clauses at locations where adverb phrases are also able to occur, which are typically clause initial or clause final positions. Integration from a containing clause happens when the containing clause is being parsed with clause_top_layer or verb_phrase_layer rules (both from section 4.2) that call adverbial in (5.19), which for the purpose of detecting the subordinate clause of a subordinate conjunction calls scon_clause of (5.20) below.

  • a finite clause (IP-ADV) (the first rule)
  • a to-infinitive clause (IP-INF) (the second rule)
  • a present participle (-ing) clause (IP-PPL) (the third rule)

5.6   Relative clauses

A relative clause occurs inside a noun phrase alongside but after the noun phrase head for which it acts as a modifier, with the noun phrase head contributing a role within the relative clause. This necessitates the wherewithal to integrate the role inside the relative clause with the external noun phrase head. The integration is possible in four ways:

(i)
with a relative pronoun of which there are various kinds, although we consider only RPRO words (which, who, whom, that; see section 2.2)
(ii)
through the complete absence of a nevertheless selected element internal to the relative clause, represented in the output parse with a trace (*T*).
(iii)
as a reduced relative clause that has the form of a participle clause occuring directly inside the noun phrase whose head is modified, so with a participle verb and its complements, but no finite verb and no clause subject.
(iv)
as a to-infinitive clause with a missing selected element

Cases (i) and (ii) are captured with the first relative_clause rule of (5.26), case (iii) with the second and third rules (which respectively call ip_ppl from section 4.4.2 and ip_ppl_passive from section 4.4.4), and case (iv) with the fourth rule (which calls ip_to_inf from section 4.4.1).

5.7   Cleft constructions

Support for cleft constructions can be added with the extra verb_complement rule of (5.41), which selects as complements a noun phrase as the focus phrase (-FOC) and a cleft clause (IP-CLF) that has the internal structure of a finite relative clause. This requires the verb code to be ;~cleft_Vn and also requires the Sbj_type parameter of the clause to be set to the value of cleft. This Sbj_type parameter setting is possible with the first subject rule in section 2.5 picking up a word with word class PRO;_clf_, which can only ever be an instance of it.
