TGrep2 Search Guide


TGrep2 is a tool for searching syntax trees. This guide provides an introduction to the TGrep2 query language. This guide sources explanatory detail from the manual that accompanies TGrep2.



1   Introduction

TGrep2 by Douglas Rohde (Rohde 2005) is a tool for searching syntax trees. This guide provides an introduction to the query language that TGrep2 uses when searching data.

    The TGrep2 query language closely resembles the query languages of TGrep by Richard Pito (Pito 1994). TGrep was the original tree-matching program that came with the Penn Treebank. TGrep2 is a reimplementation that also adds language extensions.

    TGrep2 queries are expressed as patterns that consist of expressions to match nodes and relationships defining links or negated links to other nodes. Nodes of searched trees are matched either with simple character strings, or OR'd character strings, or regular expressions (see section 2). Possible node relationships are illustrated and detailed in section 3.

    A complex node expression consists of a node expression followed by Boolean expressions of relationships to other nodes, as explained in Section 4. Also, nodes can be assigned labels and may be referred to elsewhere in the pattern by those labels (see section 5). Finally, patterns can be broken into multiple segments (see section 6).

    Typically, spaces are optional in patterns. But for readability, it is usually best to place spaces between relationships, node expressions, and other operators.

2   Specifying nodes in a TGrep2 pattern

Single tree nodes are matched in TGrep2 as follows:

  • A wildcard, specified by two underscores (__), matches any node.
  • A simple character string, such as IP-ADV, NP, ADV, Taro, etc., matches if it is the full content of the matched node.
  • A pattern can be built from simple character strings separated by pipes (|) with no spaces. Such a pattern will match a node if any of its components match the node's full content. For example, NP|PP will identify NP or PP nodes.
  • A regular expression, indicated by surrounding slashes (/), such as /^NP\b/, can be used as a description to match nodes which possibly differ in their full content. Regular expressions for TGrep2 are what Unix calls extended regular expressions.
  • If a simple character string or regular expression begins with !, the matching process will be complemented. That is, matches will turn into non-matches, and vice-versa. For example, !VB will match all nodes that are not VB, and !/^I/ will match any node that does not start with I.

2.1   Specifying nodes with TGrep2 regular expressions

Specified as a string following the syntax of section 2.2 below, a regular expression matches a node if there is a part of the node that is matched. For example, (1) matches PP, CP-THT, ADVP, etc.

(1)
/[PCV]P/

The matching of (1) involves a character class expression which matches any one of the single characters that are contained within square brackets ([, ]). A character class expression can also be used to specify a range of characters, thus [0-9] matches any single digit character. If the first character of a bracket expression is the caret symbol (^) then the whole expression is used to match a single character that is not contained within the square brackets, so [^0-9] matches any single non digit character.

    A different use of the caret symbol (^) is to anchor the regular expression to the beginning of a matched node. So, (2) will not match ADVP but will match PP and CP-THT.

(2)
/^[PCV]P/

A dollar sign ($) as the last character will anchor the regular expression to the end of a matched node. For example, (3) will not match NP-PRD2 but will match NP-PRD.

(3)
/PRD$/

Use of both the caret and dollar-sign in (4) constrains the match to only NP and PP.

(4)
/^[NP]P$/

A regular expression can also be constrained with \b to mark a word boundary. Thus, (5) will match NP and NP-PRD but will not match NPR.

(5)
/^NP\b/

    Regular expressions are case sensitive and so (6) and (7) match different nodes.

(6)
/^The$/
(7)
/^the$/

    Round brackets ((, )) are used to mark a subexpression called a block or capturing group. Such grouping is useful for isolating an expression component to then have operations imposed on it. For example, the question mark symbol (?) says to match the preceding element one or zero times. With (-TMP)? it is all of -TMP that is the preceding element. Within a capturing group the pipe symbol (|) can be used to separate subexpressions with the meaning to match either the first or the second expression. For example, -(TMP|MNR) is used to match nodes that contain either -TMP or -MNR.

    Note, you need to be particularly careful when writing node matching regular expressions that end with dollar signs. This is because you are forced to predict what the end content of a node could be. For example, suppose you wished for a node pattern that would match all bare noun phrases, that is, all noun phrase nodes that lack functional extensions. In addition to matching nodes that are simply NP, you will also want to match nodes like NP;{BOY}. However, you will not want to match either NP-SBJ or NP-SBJ;{BOY}, since these have subject (-SBJ) functional marking. A search with (8) would fail to match NP;{BOY}.

(8)
/^NP$/

    A regular expression that meets the need of matching all bare noun phrases is (9).

(9)
/^NP(;.+)?$/

Regular expression (9) starts with a caret symbol to anchor the regular expression to the beginning of a matched node with following NP content that is followed by one or zero instances of a capturing group. The capturing group involves a match for a semi-colon (;) followed by one or more characters, with the use of the plus symbol (+) to indicate one or more instances of the preceding element which is the dot symbol (.) that matches any character.

2.2   Regular expression syntax

This section describes the regular expression syntax used by TGrep2. The delimiter character (/) bounds a regular expression for matching nodes. The syntax for a regular expression e0 is given in (10):

(10)
  e3:  literal | charclass | ‘.’ | ‘^’ | ‘$’ | ‘(’ e0 ‘)’

  e2:  e3
    |  e2 REP

  REP: ‘*’ | ‘+’ | ‘?’ | ‘{n}’ | ‘{n,}’ | ‘{n,m}’

  e1:  e2
    |  e1 e2

  e0:  e1
    |  e0 ‘|’ e1

    A literal is any non-metacharacter, or a metacharacter (one of .*+?[](){}|\^$) preceded by ‘\’, or the delimiter (/) preceded by ‘\’, or \b (marking a “word” boundary, i.e., a maximal string of alphanumeric characters, including the underscore (_)).

    A charclass is a nonempty string s bracketed [s] (or [^s]); it matches any character in (or not in) s. A substring a-b, with a and b in ascending order, stands for the inclusive range of characters between a and b. In s, the metacharacters ‘-’, ‘]’, an initial ‘^’, and the regular expression delimiter must be preceded by a ‘\’; other metacharacters have no special meaning and may appear unescaped.

    A ‘.’ matches any character.

    A ‘^’ matches the beginning of a node; ‘$’ matches the end of the node.

    The REP operators match zero or more instances (*), one or more instances (+), zero or one instances (?), exactly n instances ({n}), at least n instances ({n,}), at least n and at most m instances ({n,m}), respectively of the preceding regular expression e2.

    A concatenated regular expression, e1e2, matches a match to e1 followed by a match to e2.

    An alternative regular expression, e0|e1, matches either a match to e0 or a match to e1.

    A match to any part of a regular expression extends as far as possible without preventing a match to the remainder of the regular expression.

3   Relationships between nodes

Relationships define connections between nodes. There is a complete pairing of relationships with counterparts, allowing for flexibility when constructing complex relationships (see section 4). Notable relationships are given in (11).

(11)
A << B     A dominates (is an ancestor of) B
A >> B     A is dominated by (is a descendant of) B
A < B      A immediately dominates (is the parent of) B
A > B      A is immediately dominated by (is the child of) B
A .. B     A precedes B
A ,, B     A follows B
A . B      A immediately precedes B
A , B      A immediately follows B
A $ B      A is a sister of and not equal to B
A $.. B    A is a sister of and precedes B
A $,, B    A is a sister of and follows B
A $. B     A is a sister of and immediately precedes B
A $, B     A is a sister of and immediately follows B

Any relationship can be negated by immediately preceding it with an exclamation mark (!).

    The following presents pictures grouping the relationships of (11) as forward and backward links:

(12)
C << __               (dominates / is an ancestor of)
__ >> C               (is dominated by / is a descendant of)
(13)
E >> __               (is dominated by / is a descendant of)
__ << E               (dominates / is an ancestor of)
(14)
C < __                (immediately dominates / is the parent of)
__ > C                (is immediately dominated by / is the child of)
(15)
I .. __               (precedes)
__ ,, I               (follows)
(16)
I ,, __               (follows)
__ .. I               (precedes)
(17)
I . __                (immediately precedes)
__ , I                (immediately follows)
(18)
I , __                (immediately follows)
__ . I                (immediately precedes)
(19)
F $ __                (sister)
__ $ F                (sister)
(20)
E $.. __              (sister and precedes)
__ $,, E              (sister and follows)
(21)
E $. __               (sister and immediately precedes)
__ $, E               (sister and immediately follows)

3.1   The i-th child relationships

To specify the exact child position which a node must occupy, use the i-th child relationships defined in (22):

(22)
A <i B     B is the ith child of A (i > 0) (the first child is <1)
A >i B     A is the ith child of B (i > 0) (the first child is >1)
A <-i B    B is the ith-to-last child of A (i > 0) (the last child is <-1)
A >-i B    A is the ith-to-last child of B (i > 0) (the last child is >-1)

    In these relationships, i represents a number indicating which child slot is to be considered. Positive numbers are counts from the left edge, while negative numbers are counts from the right edge. For example, (23) finds NP nodes where the sixth child is a PP.

(23)
/^NP\b/ <6 /^PP\b/

Pattern (24) finds NP nodes where the last child is an ADVP.

(24)
/^NP\b/ <-1 /^ADVP\b/

    To select nodes with specific numbers of children you can use a positive and a negative i-th child relationship. For example, to find NP nodes which dominate either eight or nine children you would use pattern (25):

(25)
/^NP\b/ <8 __ !<10 __

26   Other relationships

Other available relationships between nodes, additional to those given above in (11) and (22), are given in (26):

(26)
A == B     A and B are the same node
A <<, B    B is a leftmost descendant of A
A <<- B    B is a rightmost descendant of A
A >>, B    A is a leftmost descendant of B
A >>- B    A is a rightmost descendant of B
A <, B     B is the first child of A (synonymous with A <1 B)
A >, B     A is the first child of B (synonymous with A >1 B)
A <- B     B is the last child of A (synonymous with A <-1 B)
A >- B     A is the last child of B (synonymous with A >-1 B)
A <: B     B is the only child of A
A >: B     A is the only child of B
A <<: B    A dominates B via an unbroken chain (length > 0) of unary local trees
A >>: B    A is dominated by B via an unbroken chain (length > 0) of unary local trees

4   Boolean expressions for building complex patterns

Complex TGrep2 patterns can be constructed by specifying Boolean expressions of relationships that connect to a node. If no operator is placed between two relationships, they are assumed to be connected by an and, which can be specified for clarity with an ampersand (&). Thus, patterns (27) and (28) both match an IP node which immediately dominates a PP node and which dominates an IP node.

(27)
/^IP\b/ < /^PP\b/ << /^IP\b/
(28)
/^IP\b/ < /^PP\b/ & << /^IP\b/

The second relationship << /^IP\b/ refers to the first IP and not to the PP, and is equivalent to (29).

(29)
(/^IP\b/ < /^PP\b/) << /^IP\b/

The placement of parenthesis can be used to group a node with its relationship(s), so that (30) will match an IP which immediately dominates a PP which in turn dominates some IP.

(30)
/^IP\b/ < (/^PP\b/ << /^IP\b/)

    If a pipe (|) is placed between two relationships, only one of the relationships must be satisfied. Pattern (31) will find an IP node which immediately dominates a PP node or dominates an IP node:

(31)
/^IP\b/ < /^PP\b/ | << /^IP\b/

The (typically implicit) and binds tighter than the or. So pattern (32) will find an infinitive clause node (/^IP-INF\b/) that has an infinitive verb child (/^VB\b/) and an indirect object child (/^NP-OB2\b/) or that has an infinitive be child (/^BE\b/) and a predicative noun phrase child (/^NP-PRD2\b/).

(32)
/^IP-INF\b/ < /^VB\b/ < /^NP-OB2\b/ | < /^BE\b/ < /^NP-PRD2\b/

    To create more complex expressions, square brackets ([]) can be used to group relationships to form a complex relationship. Thus, pattern (33) finds an infinitive clause (/^IP-INF\b/) that has an infinitive verb child (/^NP-OB2\b/) or a noun phrase predicative child (/^NP-PRD2\b/), and that precedes a noun phrase subject sister (/^NP-SBJ\b/) or follows an adverb phrase /ADVP\b/ sister.

(33)
/^IP-INF\b/ [ < /^NP-OB2\b/ | < /^NP-PRD2\b/ ] [ $.. /SBJ/ | $,, /^ADVP\b/ ]

Because an expression with square brackets itself acts as a complex relationship connecting to a node, an open square bracket must immediately precede a relationship or another open square bracket that itself starts a complex relationship.

5   Labeled nodes and back links

TGrep2 allows for a node to be given a unique label and then referred to elsewhere by that label. If a node name is followed by an equal sign (=) and a label, the label will be assigned to the node. If the node name only consists of an equal sign and a label, it serves as a back link that refers to the node that has been assigned to that name elsewhere in the pattern.

    Having back links leads to the creation of patterns with cycles in the link structure. For example, imagine we want to find sentences with present tense verbs in a matrix clause that select an object that clause. With access to labels and back links, we can do this with the pattern of (34).

(34)
__=mid
   < /CP-THT-OB1/
   << (/^VBP\b/=verb !>> (/^(IP|FS)\b/ >> =mid))
   [ == /^IP-MAT\b/ | !== /^IP\b/ >> (/^IP-MAT\b/ !<< (/^IP\b/ << =mid)) ]

Pattern (34) works by stating that the node named mid is directly above the object that clause and this should also be an ancestor to the present tense verb without there being an intervening clause or FS node. Finally, it is also required that the common ancestor named mid is either itself the matrix clause or is itself not a clause node and has no intermediate clause node between it and the matrix clause.

    As another example of how back links are useful, consider the task of finding relative clauses with object traces. An object trace might be embedded arbitrarily deeply inside its relative clause, which is captured with the dominates relation (<< (/OB1/ < /\*T\*/)). However, without further qualification, this is too liberal, as it will also allow the return of a relative clause that itself has no object trace but contains a relative clause with an object trace. We can add the extra qualification to prohibit the unwelcome scenario of picking an incorrect relative clause node for the trace with a back link, resulting in (35).

(35)
/REL/=p << (/OB1/ < /\*T\*/ !>> (/REL/ >> =p))

The added back link statement (!>> (/REL/ >> =p)) requires there to be no intervening relative clause (/REL/) between the object trace and the relative clause that the pattern serves to find.

6   Segmented patterns

TGrep2 provides a way to split patterns into multiple segments, which are separated by colons (:). All segments need to be related. For example, segments following the first can begin with a reference to a labeled node that is defined in a previous segment. Thus, pattern (36) will find trees where an NP immediately dominates a CP-THT.

(36)
/^NP\b/=x : =x < /^CP-THT\b/

    With labeling of complex nodes, segments can entirely replace patterns that otherwise require embedded relationships. For example, tree fragment (37) is matched by pattern (38) without segments as well as pattern (39) with segments.

(37)
(38)
/^NP\b/ < /^N\b/ < (/^CP-THT\b/ < (/^IP-SUB\b/ < (/^C\b/ < that)))
(39)
/^NP\b/=x : (=x < /^N\b/=n) : (=x < /^CP-THT\b/=y) :
(=y < /^IP-SUB\b/=i) : (=i < /^C\b/=c) : (=c < that)

    Segments can make patterns easier to read and debug. Segments can also assist with pattern reuse. Patterns can frequently be mostly the same while differing in only one region. For such cases, the region that changes can be placed in its own segment, so that full patterns are built when a unique segment is appended to a common base, or vice versa.
